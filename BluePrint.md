\# بلوبرنت كاملة: نظام المحادثة الذكي لاستفسارات الطلبة



\---



\## أولاً: فهم المشروع بعمق قبل ما نبدأ



الخطأ اللي بيوقع فيه معظم طلاب التخرج إنهم يبدأوا يكودوا مباشرةً من غير ما يسألوا السؤال الجوهري:



\*\*"الشات بوت هذا بيجاوب على إيه بالضبط؟"\*\*



لازم صاحبك يحدد الـ scope قبل أي حاجة:

\- استفسارات القبول والتسجيل؟

\- جداول المحاضرات؟

\- نتائج الامتحانات؟

\- معلومات عن الدكاترة والأقسام؟

\- الأنشطة الطلابية؟



لأن ده هو اللي هيحدد \*\*حجم قاعدة المعرفة\*\* وبالتالي كل حاجة تانية.

\## ثانياً: القرار المعماري — الأهم في المشروع



في تلاتة مداخل للـ Chatbot:



\### ❌ المدخل الأول: Rule-based (لا تعمله)

أسئلة وإجابات ثابتة، if/else منطق. مشكلته إنه ما بيفهم اللغة الطبيعية. ده مش "ذكي" وما هيقبله أي لجنة مناقشة سنة 2025.



\### ⚠️ المدخل الثاني: Fine-tuned Model (ثقيل جداً)

تدريب موديل من الصفر — يحتاج بيانات ضخمة وحوسبة ما متاحة.



\### ✅ المدخل الثالث: RAG — Retrieval Augmented Generation

\*\*هذا هو الاختيار الصح للمشروع ده.\*\*



الفكرة ببساطة:

```

الطالب يسأل → النظام يبحث في قاعدة المعرفة → يجيب الأجزاء ذات الصلة → LLM يصيغ إجابة طبيعية

```



\*\*ليه RAG؟\*\*

\- قابل للشرح أكاديمياً بعمق (embeddings، vector search، context window)

\- سهل التحديث (تضيف وثائق جديدة بدون إعادة تدريب)

\- نتائجه دقيقة لأنه مقيد بالمعلومات اللي أنت أضفتها

\- عصري ومحترم في لجان المناقشة

\## رابعاً: الـ Tech Stack مع التبرير لكل اختيار



\### Backend: Python + FastAPI

\*\*ليه لا Flask؟\*\* FastAPI أسرع، عنده async native، وdocumentation تلقائية بـ Swagger — الأخيرة مفيدة جداً في العرض.



\### AI Layer: LangChain

Framework بيربط كل حاجة: الـ embeddings، الـ vector store، والـ LLM. بيخلي الكود منظم وأكاديمي.



\### LLM: خيارين حسب الميزانية



\*\*خيار A (مدفوع، أسهل):\*\* OpenAI GPT-4o-mini

\- تكلفة رخيصة جداً للمشروع (\~$5 تكفي)

\- نتائج ممتازة بالعربي



\*\*خيار B (مجاني تماماً):\*\* Ollama + Llama 3 / Gemma 2

\- يشتغل محلياً على اللاب توب

\- ما فيه تكلفة

\- للعرض أمام اللجنة أحسن (ما محتاج نت)

\- \*\*أنصح بيه للمشروع الأكاديمي\*\*



\### Embeddings

`sentence-transformers` مع موديل عربي: `paraphrase-multilingual-MiniLM-L12-v2`

هذا الموديل بيفهم العربي والإنجليزي مع بعض.



\### Vector Database: ChromaDB

\- مجاني، بيشتغل locally

\- لا يحتاج server منفصل

\- سهل التعامل معاه في Python



\### Frontend: React + Tailwind

أو حتى HTML/CSS/JS البسيط إذا الوقت ضيق — لكن React أحسن للـ portfolio.



\### Database للمحادثات: SQLite

بسيطة، ما تحتاج setup، وكافية تماماً.

\## سادساً: منطق RAG Pipeline بالتفصيل



\### المرحلة الأولى: بناء قاعدة المعرفة (تُعمل مرة واحدة)



```

1\. جمع الوثائق (PDFs, Word, نصوص)

&#x20;         ↓

2\. تقطيع النصوص إلى chunks

&#x20;  (كل chunk = 500 كلمة مثلاً مع 50 كلمة overlap)

&#x20;         ↓

3\. تحويل كل chunk لـ embedding (vector رقمي)

&#x20;         ↓

4\. تخزين الـ vectors في ChromaDB

```



\### المرحلة الثانية: الإجابة على سؤال (تتكرر مع كل سؤال)



```

الطالب يكتب سؤال

&#x20;         ↓

تحويل السؤال لـ embedding

&#x20;         ↓

البحث في ChromaDB عن أقرب 3-5 chunks

(Semantic Search — مش keyword search)

&#x20;         ↓

بناء الـ Prompt:

"أنت مساعد جامعي. باستخدام المعلومات التالية فقط:

\[الـ chunks المسترجعة]

أجب على سؤال الطالب: \[السؤال]

إذا المعلومة غير موجودة، قل ذلك صراحة."

&#x20;         ↓

إرسال للـ LLM → استقبال الإجابة

&#x20;         ↓

حفظ المحادثة في SQLite

&#x20;         ↓

إرسال الإجابة للطالب

```

\## سابعاً: قاعدة البيانات — Schema



```sql

\-- جدول المحادثات

CREATE TABLE conversations (

&#x20;   id TEXT PRIMARY KEY,

&#x20;   student\_id TEXT,

&#x20;   started\_at DATETIME,

&#x20;   ended\_at DATETIME,

&#x20;   message\_count INTEGER

);



\-- جدول الرسائل

CREATE TABLE messages (

&#x20;   id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   conversation\_id TEXT,

&#x20;   role TEXT,           -- 'user' أو 'assistant'

&#x20;   content TEXT,

&#x20;   timestamp DATETIME,

&#x20;   tokens\_used INTEGER,

&#x20;   retrieved\_chunks TEXT,    -- JSON: أي chunks استُخدمت

&#x20;   confidence\_score REAL,

&#x20;   FOREIGN KEY (conversation\_id) REFERENCES conversations(id)

);



\-- جدول الوثائق

CREATE TABLE documents (

&#x20;   id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   filename TEXT,

&#x20;   doc\_type TEXT,       -- 'pdf', 'txt', 'manual\_entry'

&#x20;   added\_at DATETIME,

&#x20;   chunk\_count INTEGER,

&#x20;   status TEXT          -- 'processed', 'pending', 'failed'

);



\-- جدول التقييمات (للمتابعة)

CREATE TABLE feedback (

&#x20;   id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   message\_id INTEGER,

&#x20;   rating INTEGER,      -- 1-5

&#x20;   comment TEXT,

&#x20;   timestamp DATETIME

);

```

\## ثامناً: قاعدة المعرفة — الـ Content



هذا الجزء الأهم وأكتر ناس بيتجاهلوه. الـ AI بيكون بقدر ما تعطيه معلومات.



\*\*الوثائق اللازمة (صاحبك يجمعها):\*\*



| الفئة | المحتوى |

|-------|---------|

| القبول | شروط القبول، المواعيد، الأوراق المطلوبة |

| التسجيل | إجراءات التسجيل، إضافة/حذف مادة، الرسوم |

| الأكاديمية | الخطط الدراسية، الجداول، متطلبات التخرج |

| الإدارة | أوقات الدوام، أرقام الهواتف، أماكن المكاتب |

| الأنشطة | الأندية، الفعاليات، الخدمات الطلابية |

| الـ FAQ | أكثر 50 سؤال بيسألها الطلاب |



\*\*نصيحة استراتيجية:\*\* اجمع الـ FAQ من موظفين شؤون الطلاب فعلياً — هم يعرفوا إيه الأسئلة الحقيقية.

\## تاسعاً: الـ System Prompt (قلب الشخصية)



```

أنت "نور"، المساعد الذكي لطلاب \[اسم الجامعة].



مهمتك:

\- الإجابة على استفسارات الطلاب بدقة وبشكل ودي

\- الاعتماد فقط على المعلومات الموجودة في قاعدة المعرفة

\- إذا لم تعرف الإجابة، أخبر الطالب بذلك بوضوح واقترح التواصل مع الجهة المختصة



القواعد:

1\. لا تخترع معلومات غير موجودة

2\. تكلم بالعربية الفصحى البسيطة

3\. إذا السؤال خارج نطاق الجامعة، أعتذر بلطف

4\. دائماً اختم بسؤال: هل تحتاج مزيداً من المساعدة؟



السياق المسترجع: {context}

سؤال الطالب: {question}

```

\## حادي عشر: الـ Evaluation — الجزء اللي بيفرق في الدرجات



لجان المناقشة بتسأل: "كيف قست دقة نظامك؟"



\*\*صاحبك لازم يقيس:\*\*



```

1\. Retrieval Accuracy

&#x20;  هل النظام بيجيب الـ chunks الصحيحة؟

&#x20;  → اعمل 50 سؤال تعرف إجابتها، شوف كم مرة جاب المعلومة الصح



2\. Answer Relevance

&#x20;  هل الإجابة ذات صلة بالسؤال؟

&#x20;  → تقييم يدوي (1-5) من عينة طلاب



3\. Response Time

&#x20;  كم ثانية الإجابة؟

&#x20;  → قياس متوسط وقت الاستجابة



4\. Coverage Rate

&#x20;  كم نسبة الأسئلة اللي قدر يجاوب عليها؟

&#x20;  → الأسئلة اللي قال "لا أعرف" فيها



المقارنة الذكية:

قارن بين:

\- RAG فقط

\- RAG + conversation history

\- بدون RAG (LLM مباشر)

```



ده بيعطيه \*\*contribution حقيقية\*\* للبحث.

\## ثاني عشر: ما يتجنبوه (بناءً على أخطاء شائعة)



1\. \*\*لا تعمل authentication معقد\*\* — مضيعة وقت، يكفي session ID بسيط

2\. \*\*لا تحاول تدعم voice\*\* — scope زيادة

3\. \*\*لا تستخدم GPT-4 في العرض\*\* — تكلفة وانقطاع نت. استخدم Ollama محلياً

4\. \*\*لا تهمل الـ chunk size\*\* — ده بيأثر على الدقة تأثيراً كبيراً. جرب 300، 500، 800

5\. \*\*لا تنسى الـ conversation context\*\* — النظام لازم يتذكر آخر 3-4 رسائل علشان يفهم الـ follow-up questions

6\. \*\*لا تبدأ بالـ frontend\*\* — ابدأ بالـ pipeline في terminal وتأكد إنه شغال زين



\---



\## ثالث عشر: نقاط القوة الأكاديمية اللي يبرزها



\- \*\*الـ Hallucination Prevention:\*\* النظام مقيد بالمعلومات المعطاة، مش بيخترع

\- \*\*Explainability:\*\* يعرف يقول "هذه الإجابة مأخوذة من وثيقة X"

\- \*\*Scalability:\*\* تضيف وثائق جديدة بدون تغيير الكود

\- \*\*Arabic Language Support:\*\* كثير أنظمة ضعيفة في العربي

\- \*\*Privacy:\*\* إذا استخدم Ollama، البيانات ما بتطلع برا الجامعة
